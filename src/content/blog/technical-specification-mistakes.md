# 10 помилок при створенні технічного завдання

*Автор: Марія Коваленко • Дата: 12 січня 2024 • Час читання: 12 хв*

![Технічне завдання](https://images.pexels.com/photos/3184291/pexels-photo-3184291.jpeg?auto=compress&cs=tinysrgb&w=800)

Технічне завдання (ТЗ) — це фундамент успішного проекту. Погано написане ТЗ призводить до непорозумінь, затримок, перевитрат бюджету та незадоволення обох сторін. За роки роботи з різними клієнтами я зібрала топ-10 найпоширеніших помилок при створенні технічного завдання та способи їх уникнення.

## Помилка #1: Відсутність чіткої мети проекту

### Що не так:
```
❌ "Нам потрібен сайт для нашої компанії"
❌ "Хочемо як у конкурентів, тільки краще"
❌ "Зробіть щось сучасне та красиве"
```

### Як правильно:
```
✅ "Потрібен корпоративний сайт для збільшення кількості 
   заявок на консультації з юридичних питань на 30% 
   протягом 6 місяців"

✅ "Інтернет-магазин для продажу handmade прикрас з 
   можливістю онлайн-оплати та доставки по Україні"
```

### Чому це важливо:
- Розробники розуміють, що саме потрібно створити
- Можна виміряти успішність проекту
- Легше приймати рішення під час розробки

### Шаблон опису мети:
```
Мета: [Що потрібно досягти]
Цільова аудиторія: [Хто буде користуватися]
Ключові метрики: [Як вимірювати успіх]
Терміни: [Коли потрібен результат]
```

## Помилка #2: Розпливчатий опис функціоналу

### Що не так:
```
❌ "Сайт повинен бути зручним"
❌ "Потрібна система авторизації"
❌ "Додайте можливість залишати відгуки"
```

### Як правильно:
```
✅ Система авторизації:
   - Реєстрація через email з підтвердженням
   - Вхід через email/пароль
   - Відновлення паролю через email
   - Можливість входу через Google/Facebook
   - Запам'ятовування сесії на 30 днів

✅ Система відгуків:
   - Тільки авторизовані користувачі можуть залишати відгуки
   - Рейтинг від 1 до 5 зірок
   - Текстовий коментар до 500 символів
   - Можливість додавати фото (до 3 штук)
   - Модерація відгуків адміністратором
```

### Чек-лист для опису функціоналу:
- [ ] Хто може користуватися функцією?
- [ ] Які дії можна виконувати?
- [ ] Які обмеження існують?
- [ ] Як функція взаємодіє з іншими?
- [ ] Які повідомлення показувати користувачу?

## Помилка #3: Ігнорування технічних вимог

### Що не так:
Взагалі не вказувати технічні вимоги або писати:
```
❌ "Використовуйте сучасні технології"
❌ "Сайт повинен швидко працювати"
❌ "Зробіть адаптивним"
```

### Як правильно:
```
✅ Технічні вимоги:

ПРОДУКТИВНІСТЬ:
- Час завантаження головної сторінки: до 3 секунд
- Час завантаження внутрішніх сторінок: до 2 секунд
- Підтримка одночасно 1000 користувачів

АДАПТИВНІСТЬ:
- Коректне відображення на екранах від 320px до 1920px
- Тестування на iPhone, Android, планшетах
- Підтримка touch-жестів

БРАУЗЕРИ:
- Chrome (останні 2 версії)
- Firefox (останні 2 версії)  
- Safari (останні 2 версії)
- Edge (остання версія)

БЕЗПЕКА:
- SSL сертифікат
- Захист від SQL-ін'єкцій
- Захист від XSS атак
- Регулярні бекапи (щоденно)

SEO:
- Семантична верстка
- Meta теги для всіх сторінок
- Sitemap.xml
- Robots.txt
- Швидкість завантаження 90+ за Google PageSpeed
```

## Помилка #4: Відсутність дизайн-вимог

### Що не так:
```
❌ "Дизайн на ваш розсуд"
❌ "Щось в стилі Apple"
❌ "Використовуйте наші корпоративні кольори"
```

### Як правильно:
```
✅ Дизайн-вимоги:

СТИЛЬ:
- Мінімалістичний, сучасний дизайн
- Акцент на зручності використання
- Професійний, але дружній тон

КОЛЬОРИ:
- Основний: #2563EB (синій)
- Додатковий: #10B981 (зелений)  
- Акцентний: #F59E0B (помаранчевий)
- Текст: #1F2937 (темно-сірий)
- Фон: #F9FAFB (світло-сірий)

ТИПОГРАФІКА:
- Заголовки: Inter, жирність 600-700
- Основний текст: Inter, жирність 400
- Розмір основного тексту: 16px
- Міжрядковий інтервал: 1.5

ЕЛЕМЕНТИ:
- Кнопки з заокругленими кутами (8px)
- Тіні для карток та модальних вікон
- Плавні анімації переходів (300ms)
- Іконки в стилі Heroicons або Feather

РЕФЕРЕНСИ:
- [Посилання на сайти-приклади]
- [Мудборд з Pinterest]
```

## Помилка #5: Нечіткі вимоги до контенту

### Що не так:
```
❌ "Контент надамо пізніше"
❌ "Використовуйте Lorem Ipsum"
❌ "Візьміть тексти з нашого старого сайту"
```

### Як правильно:
```
✅ Контент-план:

ТЕКСТИ:
- Головна сторінка: готовий (надається до 15.01)
- Про компанію: потрібна допомога з написанням
- Послуги: є чернетки, потрібне редагування
- Контакти: готовий

ЗОБРАЖЕННЯ:
- Логотип: є у векторному форматі
- Фото команди: будемо робити фотосесію
- Фото продукції: є 20 якісних фото
- Іконки: потрібно підібрати в єдиному стилі

ВІДЕО:
- Презентаційне відео: планується зйомка
- Відгуки клієнтів: є 5 відео, потрібен монтаж

ДОКУМЕНТИ:
- Прайс-лист: оновлюється щомісяця
- Сертифікати: є PDF файли
- Презентації: потрібно адаптувати для веб
```

### Чек-лист контенту:
- [ ] Хто відповідає за створення контенту?
- [ ] Які терміни надання матеріалів?
- [ ] В якому форматі надавати файли?
- [ ] Чи потрібна допомога з копірайтингом?
- [ ] Які вимоги до якості зображень?

## Помилка #6: Неврахування інтеграцій

### Що не так:
Забувати про необхідні інтеграції або згадувати про них в останній момент.

### Як правильно:
```
✅ Необхідні інтеграції:

ПЛАТІЖНІ СИСТЕМИ:
- LiqPay (основний)
- Portmone (резервний)
- Можливість додавання нових систем

ДОСТАВКА:
- Нова Пошта API
- УкрПошта API
- Самовивіз з точок видачі

АНАЛІТИКА:
- Google Analytics 4
- Facebook Pixel
- Hotjar для аналізу поведінки

CRM:
- Інтеграція з AmoCRM
- Автоматична передача заявок
- Синхронізація статусів замовлень

EMAIL МАРКЕТИНГ:
- MailChimp для розсилок
- Автоматичні email після покупки
- Нагадування про кинуті кошики

СОЦІАЛЬНІ МЕРЕЖІ:
- Кнопки поширення в соцмережах
- Віджет Instagram
- Чат-бот Telegram для підтримки
```

## Помилка #7: Відсутність сценаріїв використання

### Що не так:
Описувати тільки функції, але не показувати, як користувач буде ними користуватися.

### Як правильно:
```
✅ Сценарій: Покупка товару

1. ПОШУК ТОВАРУ:
   Користувач заходить на сайт → використовує пошук або 
   каталог → переглядає картку товару

2. ДОДАВАННЯ В КОШИК:
   Обирає характеристики → вказує кількість → 
   натискає "Додати в кошик" → бачить повідомлення про успіх

3. ОФОРМЛЕННЯ ЗАМОВЛЕННЯ:
   Переходить в кошик → перевіряє товари → 
   натискає "Оформити замовлення"

4. ЗАПОВНЕННЯ ДАНИХ:
   Вказує контактні дані → обирає спосіб доставки → 
   обирає спосіб оплати → підтверджує замовлення

5. ПІДТВЕРДЖЕННЯ:
   Отримує номер замовлення → отримує email з деталями → 
   може відстежувати статус в особистому кабінеті

АЛЬТЕРНАТИВНІ СЦЕНАРІЇ:
- Що якщо товару немає в наявності?
- Що якщо користувач передумав?
- Що якщо платіж не пройшов?
```

## Помилка #8: Нереалістичні терміни та бюджет

### Що не так:
```
❌ "Потрібно зробити за тиждень"
❌ "Бюджет обмежений, але функціонал великий"
❌ "Терміни критичні, але ТЗ ще не готове"
```

### Як правильно:
```
✅ Планування проекту:

ЕТАПИ РОЗРОБКИ:
1. Аналіз та планування (1 тиждень)
2. Дизайн макетів (2 тижні)
3. Верстка (2 тижні)
4. Програмування (3 тижні)
5. Тестування та правки (1 тиждень)
6. Наповнення контентом (1 тиждень)

ЗАГАЛЬНИЙ ТЕРМІН: 10 тижнів

БЮДЖЕТ ПО ЕТАПАХ:
- Дизайн: $800
- Верстка: $600
- Програмування: $1200
- Тестування: $200
- Інтеграції: $400

ЗАГАЛЬНИЙ БЮДЖЕТ: $3200

РИЗИКИ:
- Затримка надання контенту (+1 тиждень)
- Зміни в ТЗ під час розробки (+20% до бюджету)
- Складнощі з інтеграціями (+1 тиждень)
```

### Формула розрахунку термінів:
```
Базовий термін + 20% на непередбачені ситуації + 
час на правки та тестування = Реальний термін
```

## Помилка #9: Ігнорування підтримки та розвитку

### Що не так:
Думати тільки про запуск, забуваючи про те, що буде після.

### Як правильно:
```
✅ План підтримки:

ТЕХНІЧНА ПІДТРИМКА:
- Гарантійний період: 3 місяці
- Виправлення критичних помилок: 24 години
- Виправлення некритичних помилок: 72 години
- Консультації по роботі з сайтом: безкоштовно

ХОСТИНГ ТА ДОМЕН:
- Хто відповідає за продовження?
- Де зберігаються паролі доступу?
- Як часто робити бекапи?

ОНОВЛЕННЯ:
- Оновлення CMS та плагінів: щомісяця
- Оновлення контенту: хто відповідає?
- Додавання нового функціоналу: окремо

АНАЛІТИКА ТА ОПТИМІЗАЦІЯ:
- Щомісячні звіти по відвідуваності
- Рекомендації по покращенню конверсії
- A/B тестування ключових елементів

РОЗВИТОК:
- План розвитку на наступний рік
- Бюджет на нові функції
- Пріоритети покращень
```

## Помилка #10: Відсутність критеріїв приймання

### Що не так:
```
❌ "Зробіть як краще"
❌ "Побачимо що вийде"
❌ "Головне щоб працювало"
```

### Як правильно:
```
✅ Критерії приймання:

ФУНКЦІОНАЛЬНІСТЬ:
- Всі описані функції працюють без помилок
- Форми відправляють дані та показують підтвердження
- Пошук видає релевантні результати
- Фільтри працюють коректно

ДИЗАЙН:
- Макети реалізовані з точністю 95%
- Всі елементи відображаються коректно
- Анімації працюють плавно
- Кольори відповідають брендбуку

ПРОДУКТИВНІСТЬ:
- Швидкість завантаження відповідає вимогам
- Сайт працює стабільно під навантаженням
- Немає помилок в консолі браузера

АДАПТИВНІСТЬ:
- Коректне відображення на всіх пристроях
- Зручність використання на мобільних
- Всі функції доступні на планшетах

БЕЗПЕКА:
- Пройдено базове тестування на вразливості
- Налаштовано SSL
- Форми захищені від спаму

ДОКУМЕНТАЦІЯ:
- Інструкція для адміністратора
- Опис всіх функцій
- Контакти для технічної підтримки
```

## Висновки

Якісне технічне завдання — це інвестиція в успіх проекту. Витративши час на детальний опис вимог, ви:

1. **Заощадите гроші** — менше правок та переробок
2. **Прискорите розробку** — розробники точно знають що робити
3. **Отримаете очікуваний результат** — без неприємних сюрпризів
4. **Уникнете конфліктів** — всі домовленості зафіксовані

**Пам'ятайте**: краще витратити день на написання ТЗ, ніж тиждень на виправлення помилок.
